数据传输编码

一个数据传输编码的例子：

语句 正向动作 反向动作
append “string” 追加字符串“string” 去掉最后“string”长度的字符串
base64 base64编码 base64解码
base64url URL-safe base64编码 URL-safe base64解码
mask 随机key进行XOR mask w 使用相同的key进行XOR mask w
netbios netbios 编码‘a’ netbios解码‘a’
netbiosu netbios解码’A’ netbios解码‘A’
prepend “string” 开头插入“string” 从开头去掉“string”长度的字符串

一次数据传输可以以任意次序使用上述任意个传输编码指令，例如你可以先用netbios编码一下，然后加点东西，最后用base64编码编码整个数据部分。也就是你的语句先后顺序会影响数据的处理。

数据传输部分必须以一个告诉客户端和服务端传送的数据在什么位置的指令为结束。以下有四个结束语句。

语句 意义
header “header” 把数据放到名为header的http头中
parameter “key” 把数据放到名为key的uri参数中
print 把数据放到http响应body中
uri-append 把数据直接追加到URI上

值得注意的是，print语句只能用在服务端的输出中，也就是只能用在http-get.server.output、http-post.server.output和http-stager.server.output的配置块中。其他的语句你可以任意使用。

如果你同时使用了header、parameter和 uri-append在httppost.client.output配置块中， beacon会合理划分响应的数据，适应这种传输方式。

cobalt strike的profile多个地方支持自定义的字符串，同时他还支持如下的特殊字符串格式。

输入的值 解析出的值
“\n” 换行符
“\r” 回车
“\t” tab
“\u###” unicode字符串
“\x##” 一个字节
“\\“ 反斜线

一次beacon通信的全过程

  • beacon携带metadata以GET方法请求Cobalt Strike web 服务器。（metadata是一段加密的数据，但是没有编码，所以他不能在请求头和URI中发送。需要将其所使用base64、base64url或者netbios编码之后才能在请求头和URI中发送。）
  • cs web服务器响应该请求，返回一块加密的二进制数据，你可以在http-get的output的配置块底下进行配置，对它进行编码。当所有任务完成，客户端会查看是否有输出，如果没有输出，beacon开始睡眠，如果有输出，则开始准备开始进行POST请求。
  • HTTP POST请求包含通过URI参数传递的id参数，用来标识，该输出是哪个beacon的，POST请求携带的内容也是一块加密的二进制数据，但是没有进行编码，你可以通过http-post下的client的output配置块进行配置。
  • cs服务器也许需要响应POST请求，beacon不会取用响应内容，你可以通过http-post下的server下的output配置块进行配置。

默认情况下http-get使用GET方法，http-post使用POST方法，但是你可以通过verb配置项去改变它。

通信过程中客户端和服务器发送接收的数据，或者说这是profile可自定义的项目。

请求 主体 数据块 数据内容
http-get client metadata session的metadata
http-get server output beacon的任务
http-post client id 完成任务的session id
http-post client output beacon完成任务的结果
http-post server output 空
http-stager server output 编码的payload stage

HTTP Staging

beacon是通过stager下载到内存中去的。在beacon注入目标内存之前，http-get和http-post都不会生效。你可以自己定义stager下载stage的行为。

HTTP响应头

http-config可以配置所以的cs服务器的相应头。优先级最低，相当于配置一个默认配置。set headers设置http头的顺序，不在该列表中的，就被加到最后。

内存混淆免杀的全过程

使用prepend配置可以绕过只检测前几个字节来发现内存注入行为的AV。如果是检测特定字符串的AV，就可以使用strrep配置来绕过。

如果靠strrep还绕不过，那就使用sleep_mask，让beacon在睡眠之前混淆自己。睡眠结束之后，就反混淆，并且进行正常请求，每当beacon在睡眠或者等他的父session的指令时，都会混淆自己。

轻度混淆使用stomppe，重度混淆使用obfuscate。

默认情况下，beacon会使用VirtualAlloc申请内存，然后写入dll。但是也可以使用内存模块覆盖的方式去写入dll。将module_x86设置成一个大约比beacon payload大两倍的dll，beacon将会加载它，然后在内存中找到他的地址，然后覆盖它。module_x64也是类似的。

如果怕内存中stage被检测出来，可以设置cleanup为true，这样的话，当stage没用了之后就释放掉。

进程注入行为定义

transform-x86和transform-x64定义beacon注入的内容，支持prepend和append两个配置。要确保注入的内容是可以在x86或者x64的机器上运行的代码，c2init并不会去检查插入的内容是否合法。

disable配置用了禁用一些在dll注入中使用的函数。禁用这些函数可能会引起可避免的错误，c2init会提示这些错误。

其他配置：

选项 示例 描述
min_alloc 4096 当注入时，最少申请多少内存。
startrwx false 是否使用rwx作为代码内存的默认权限，替代权限是rw
userwx false 同上，但是替代权限是rx
